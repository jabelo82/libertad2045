  6 fixes aplicados + análisis de rebalanceo en producción + validación backtest

  FIXES APLICADOS:

  C-1  chmod 600 .env en torre y VPS (aplicado manualmente en sesión)
       Archivo .env contiene TELEGRAM_TOKEN y GITHUB_TOKEN — permisos
       restrictivos son obligatorios antes de LIVE.

  C-2  SIGTERM handler en libertad2045.py — apagado limpio tras timeout:
       run_bot.sh usa `timeout 3600` que envía SIGTERM, lo que en CPython
       bypassea los bloques `finally`. El handler captura SIGTERM y lanza
       `SystemExit`, que sí propaga a través de `finally` → lockfile liberado
       + conexión IB cerrada limpiamente.

  C-4  Stop del rebalanceo usa calcular_trailing_stop() con TRAILING_FACTOR=0.75:
       El bug anterior usaba calcular_posicion() que devuelve la distancia bruta
       ATR×mult sin aplicar TRAILING_FACTOR=0.75. Cada reemplazo de stop por
       rebalanceo dejaba el stop un ~25% más ancho que el modelo. Corregido
       llamando directamente a calcular_trailing_stop(df) que aplica el factor.

  A-3  Validar frescura de datos (máx 5 días) en portfolio_manager y rebalance:
       Si la última barra de precios tiene >5 días de antigüedad, el stop no se
       evalúa ni actualiza (evita usar datos stale de festivos o gaps de datos).

  A-4  Detectar stops GTC duplicados en portfolio_manager y rebalance:
       Si IBKR tiene dos stops GTC activos para el mismo símbolo, se registra
       CRITICAL en log + alerta Telegram crítica. Se conserva el stop con precio
       más alto (mayor protección).

  A-5  Verificar estado del nuevo stop GTC antes de cancelar el anterior:
       En _reemplazar_stop_gtc(), después de placeOrder() se verifica el estado
       del nuevo stop (sleep 1s). Si el estado es Inactive/Cancelled/Rejected,
       se aborta y se conserva el stop anterior. El sistema no queda sin
       protección por un fallo en el reemplazo.

  fix  git_backup detecta cambios staged en lugar de working tree:
       git status --porcelain detectaba dashboard.html (working tree) y el bot
       intentaba commitear sin staged → git commit fallaba silenciosamente.
       Corregido con git diff --cached --quiet + returncode == 0.

  feat watchdog verifica git_backup del ciclo nocturno anterior:
       A las 12:00, el watchdog lee el CSV del día hábil anterior y busca la
       línea "Git backup falló". Reporta OK/FALLO/sin dato en el informe de
       Telegram del mediodía. Función check_git_backup() en watchdog.py.

10. INCIDENTE Y FIXES — 20/06/2026
--------------------------------------------------------------------------------

  BUG CRÍTICO — BOT NO CORRIÓ LA NOCHE DEL 19/06 (FESTIVO JUNETEENTH):

    Causa:   Regresión del commit b70c864 (fix C-2, SIGTERM handler).
             La variable de bucle `for signal in signals:` en libertad2045.py
             sombreaba el módulo `signal` importado. Python trata `signal` como
             variable local de toda la función main(), causando UnboundLocalError
             al ejecutar signal.signal(signal.SIGTERM, ...) en la línea 236.
    Fix:     Variable de bucle renombrada a `senal` (líneas 528-531).
             Commit 9e9eaba
    Nota:    El sistema no tiene lógica de calendario de festivos USA (solo
             filtra L-V). El bot intentó correr el 19/06 (viernes) sin saber
             que era festivo — pendiente a considerar.

  INCIDENTE — IB GATEWAY DE LA TORRE CAÍDO POR SESIÓN DUPLICADA:

    Causa:   Apertura de la app móvil de IBKR con las mismas credenciales
             disparó el escenario 6 de IBC ("Existing session detected"),
             cerrando el Gateway con exit code 0.
             Como Restart=on-failure solo reacciona a exit code ≠0, el servicio
             no se relanzó automáticamente.
    Fix:     Restart=on-failure → Restart=always en ibgateway.service,
             aplicado en torre y VPS. Commit 83ac75c (docs actualizados).
    Lección: Evitar abrir la app/web de IBKR con las mismas credenciales
             mientras el bot opera en torre o VPS.

  BUG — git_backup FALLABA POR CONFLICTO CON PUBLISHER DE GITHUB PAGES:

    Causa:   publicar_dashboard() sube index.html vía API de GitHub creando
             un commit directo en origin/main. Cuando git_backup() intentaba
             el push justo después, el remote estaba 1 commit por delante →
             reject de fast-forward.
    Fix:     Añadido `git pull --rebase origin main` entre el commit y el push
             en git_backup(). Commit 12f26f1

  VERIFICACIÓN END-TO-END (forzado manualmente, sábado sin mercado):

    Bypass temporal de ventana horaria: RISK_HOUR_START=0 RISK_HOUR_END=23
    (solo para la prueba puntual — no persistido en ningún archivo).
